# Callee agent guidance

- Preserve the CLI-only surface: `callee prompt --role <role> --message <task>`, `callee list`, `callee doctor`, and `callee setup`.
- Preserve flat role frontmatter. Do not add nested provider configuration.
- Do not add Gemini support.
- Do not add a server transport, Callee thread store, or handle binding without an explicit product decision.
- Keep role output on stdout and diagnostics on stderr.
- Use Norma Runtime for ACP process logic rather than duplicating it.
- Run `go test ./...` and `go test -race ./...` before completion.

## Release cycle

Releases are tag-triggered. Never create or push a release tag before the
release commit has passed both local and remote checks.

1. Run the local quality gate: `go test ./...`, `go test -race ./...`, the
   configured linter, and relevant plugin validation.
2. Commit the release changes and push the commit to `main` without a tag.
3. Wait for every required GitHub Actions run triggered by that commit to pass
   (including test, lint, and security workflows). Use `gh run` to inspect and
   watch the exact runs; do not infer their result from a local build.
4. Only then create and push the annotated `vX.Y.Z` tag. The tag-triggered
   release workflow performs publication.
5. Watch the tag-triggered release workflow to completion and verify that the
   tag points to the checked commit.

Do not manually publish a package as a substitute for the tag workflow, and do
not retag a commit that has not passed the remote quality gate.

## Beads Issue Tracker

Use Beads (`bd`) for durable task tracking in repositories that include it. Use the `beads` skill at `.agents/skills/beads/SKILL.md` (project install) or `~/.agents/skills/beads/SKILL.md` (global install) for Beads workflow guidance, then use the `bd` CLI for issue operations.

### Quick Reference

```bash
bd ready                # Find available work
bd show <id>            # View issue details
bd update <id> --claim  # Claim work
bd close <id>           # Complete work
bd prime                # Refresh Beads context
```

### Rules

- Use `bd` for all task tracking; do not create markdown TODO lists.
- Run `bd prime` when Beads context is missing or stale. Codex 0.129.0+ can load Beads context automatically through native hooks; use `/hooks` to inspect or toggle them.
- Keep persistent project memory in Beads via `bd remember`; do not create ad hoc memory files.

**Architecture in one line:** issues live in a local Dolt DB; sync uses `refs/dolt/data` on your git remote; `.beads/issues.jsonl` is a passive export. See https://github.com/gastownhall/beads/blob/main/docs/SYNC_CONCEPTS.md for details and anti-patterns.
